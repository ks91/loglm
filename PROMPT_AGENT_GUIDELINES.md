# Prompt Agent Developer Guidelines

This guide assumes prompt files are often generated by coding agents.
Write source content so the agent can produce a stable, readable `.md` without over-literal interpretation.

Install target:

```bash
loglm agent install <owner/repo>
```

## 1. Requirement Levels

Use RFC-style keywords:

- MUST: required for compatibility and predictable installation behavior.
- SHOULD: strongly recommended to reduce misgeneration and drift.
- MAY: optional.

## 2. Files and Resolution

Place one or more files at repository root:

- SHOULD provide `AGENT_INSTALL.md` as the shared install source for all coding agents.
- MAY provide `AGENT_INSTALL_CODEX.md` for codex-specific behavior.
- MAY provide `AGENT_INSTALL_CLAUDE.md` for claude-specific behavior.
- MAY provide `AGENT_INSTALL_GEMINI.md` for gemini-specific behavior.

Resolution order:

- codex: `AGENT_INSTALL_CODEX.md` -> `AGENT_INSTALL.md`
- claude: `AGENT_INSTALL_CLAUDE.md` -> `AGENT_INSTALL.md`
- gemini: `AGENT_INSTALL_GEMINI.md` -> `AGENT_INSTALL.md`
- `loglm` reads only the first existing file in this order for each agent (no merge).

Install output model:

- `loglm agent install <owner/repo>` stores fetched content into `<REPO-NAME-UPPER>.md` in the current directory.
- `AGENTS.md` / `CLAUDE.md` / `GEMINI.md` receive managed reference blocks that point to `<REPO-NAME-UPPER>.md`.
- Prompt-agent authors SHOULD design content assuming it will be consumed from that shared local file.

## 3. Authoring Model (Agent-Generated Output)

Assume a coding agent will read this guideline and generate/update the final prompt file.

Use strict, direct rules:

- MUST write requirements as explicit constraints.
- MUST NOT rely on implied intent.
- MUST NOT include ambiguous prose like "do this naturally" or "as needed".

Avoid instructions that cause literal noise:

- MUST NOT ask the generated file to mention its own filename.
- SHOULD NOT require references to unrelated local files.
- MUST NOT include meta text about "how this file was generated".

## 4. Required First Section

Start every prompt file with this heading exactly:

```md
## Non-Negotiable Rules
```

This section should contain short, high-priority rules that always apply.
Keep it compact and unambiguous.

## 5. Keep Markdown Coherent

The generated document should read as one self-contained markdown file.

Requirements:

- MUST use consistent heading hierarchy.
- MUST avoid broken cross-references.
- MUST avoid unresolved placeholders.
- SHOULD keep line wrapping and bullets readable.

## 6. Multi-Agent Sync Rules

Define when `CLAUDE.md` and `GEMINI.md` should mirror `AGENTS.md`.

- SHOULD keep a single shared core policy in `AGENT_INSTALL.md`.
- MAY maintain agent-specific files when tool usage, interaction style, or platform notes differ.
- MUST keep `## Non-Negotiable Rules` semantically equivalent across variants.
- MUST document intentional differences in `README.md` (short diff summary is enough).

## 7. Design for Layered Composition

`loglm` updates repository-managed blocks in local prompt files.
Your content must coexist with other prompt-agent blocks.

- MUST define your scope clearly.
- MUST NOT conflict with project-wide safety policies.
- SHOULD prefer additive behavior over replacement behavior.

## 8. Avoid `loglm` Marker Collisions

`loglm` uses markers like:

- `<!-- loglm:begin ... -->`
- `<!-- loglm:end ... -->`

Do not include these exact marker patterns in distributed prompt files.

## 9. Platform Assumptions

`loglm` injects platform notes separately (macOS, WSL2, Ubuntu on Lima, and others).
Keep prompt files portable unless platform-specific behavior is essential.

If platform-specific commands are necessary:

- SHOULD provide alternatives (`open`, `wslview`, Linux-native tools).
- MUST state selection conditions briefly and explicitly.

## 10. Encoding Policy

Use UTF-8 without BOM.

- MUST encode prompt files as UTF-8 (no BOM).
- SHOULD avoid mixed line endings.

Example checks:

```bash
file -I AGENT_INSTALL.md
iconv -f UTF-8 -t UTF-8 AGENT_INSTALL.md >/dev/null
```

## 11. Make Behavior Verifiable

- MUST include concrete workflow steps.
- SHOULD include executable command examples.
- MUST include validation checks.
- MUST include safety boundaries.

Avoid policy-only text with no operational guidance.

## 12. Runtime Requirements Model

Prompt-agent files SHOULD define runtime prerequisites explicitly so coding agents can execute setup consistently.

Recommended terms:

- Requirements: mandatory tools/runtime needed to execute the workflow.
- Optional Dependencies: non-mandatory tools that improve productivity.
- Platform Integrations: OS-specific tools and behaviors (viewer apps, path bridging, launch commands).

MUST include for each required item:

- name
- purpose ("why needed")
- check command
- install command(s) by platform
- verification command

SHOULD include locale/toolchain policy when behavior differs (for example, Japanese LaTeX policy such as `platex` + `dvipdfmx`).

## 13. Preflight Behavior (for Coding Agents)

When a prompt-agent is loaded, coding agents SHOULD run a short preflight:

- check Requirements first
- report missing items clearly
- ask for user consent before installation
- install only approved items
- run verification commands and report results

Prompt files SHOULD define this preflight flow explicitly.

## 14. Bad vs Good Examples

Filename self-reference:

- Bad: "In this AGENTS.md file, follow AGENTS.md rules first."
- Good: "Follow the rules in this document first."

Ambiguous instruction:

- Bad: "Do this naturally based on context."
- Good: "Before any command, state one-line intent and expected result."

Meta generation text:

- Bad: "This file was generated by an AI assistant from README."
- Good: Omit generation provenance from runtime prompt content.

## 15. Compliance Checklist

Before publishing, verify:

- [ ] `## Non-Negotiable Rules` exists and is first section.
- [ ] `AGENT_INSTALL.md` is present (unless intentionally agent-specific only).
- [ ] Agent-specific files use `AGENT_INSTALL_CODEX.md` / `AGENT_INSTALL_CLAUDE.md` / `AGENT_INSTALL_GEMINI.md` naming.
- [ ] No unresolved placeholders remain.
- [ ] No `<!-- loglm:begin` or `<!-- loglm:end` markers are present.
- [ ] No unnecessary filename self-references exist.
- [ ] No ambiguous phrases like "as needed" or "naturally" remain.
- [ ] AGENT_INSTALL/AGENTS/CLAUDE/GEMINI core rules are aligned (or differences documented).
- [ ] File encoding is UTF-8 without BOM.
- [ ] Requirements/Optional Dependencies/Platform Integrations are documented.
- [ ] Each required item has check/install/verify commands.
- [ ] Preflight behavior (check -> consent -> install -> verify) is documented.

## 16. Recommended Template

```md
# <Agent Name>

## Non-Negotiable Rules
- Absolute rule 1.
- Absolute rule 2.

## Purpose
- What this prompt-agent is for.

## Scope
- What tasks it should handle.
- What tasks it should not handle.

## Workflow
- Step-by-step process.

## Commands
- Frequently used commands and expected outcomes.

## Requirements
- name:
  purpose:
  check:
  install:
  verify:

## Optional Dependencies
- name:
  purpose:
  check:
  install:
  verify:

## Platform Integrations
- macOS:
- WSL2:
- Linux:

## Preflight
- Check missing Requirements first.
- Ask for consent before install.
- Install only approved items.
- Verify and report.

## Validation
- How to verify outputs.

## Safety
- Rules that must not be violated.
```
